    Stacks and Queues - Notes¶

    CSC205 - Module 7¶

    Stacks¶

    A stack is a linear data structure whose elements are added in âlast in, first outâ (LIFO) order. So, when we remove an element from a stack it will be the last one we added.

    Stack Operations¶

    • push(E element) adds an element to top of the stack.
    • pop() removes & returns the top element from the stack.
    • peek() returns the element at top of the stack.
    • size() returns the number of elements in the stack.
    • isEmpty() returns true if the stack is empty, false if it is not.

    Applications of Stacks¶

    Stacks have a wide variety of applications such as implementing an undo feature, storing the history in a browser (used by the back button), validating a string with a recursive structure (such as HTML).

    Application: Postfix Expressions¶

    One natural application of a stack is for evaluating postfix expressions. A postfix expression is written with the operator following the two operands, as opposed to the infix notation you may be used to where the operator is between the two operands.

    The following expressions are equivalent:

    Postfix: 15 8 - Infix: 15 - 8

    The benefit of postfix expressions is that they are unambiguous. They completely eliminate the need for parenthesis.

    The following expressions are equivalent:

    Postfix: 3 4 * 2 5 + - 4 * 2 / Infix: (3 * 4 - (2 + 5)) * 4 / 2

    We can use a stack to evaluate postfix expressions using the following algorithm: Scan the expression left to right

    • When we read an operand, push it on the stack.
    • When we read an operator, pop the top two elements of the stack, perform the operation, & push the result.
    • The result is the value on the stack when the expression is exhausted.

    The postfix expressions video has a walkthrough of the algorithm.

    Java API: Stacks¶

    java.util.Stack is the Java API implementation of the Stack abstract data type. It implements all the operations, but there is no interface. It extends the Vector class so it allows several non-stack operations, which can be an issue in applications where you want to only allow stack operations. (Side note: one of the benefits of limiting the interface of a data structure is that it makes debugging easier. You donât have to worry about things getting stuck in the middle of a stack if you only have push & pop operations). The Java API also provides other data structures that can perform like a stack (java.util.Deque, linked-list, etc), but again these do not use the standard stack method names and it is up to the programmer to ensure that the data structure acts like a stack.

    We can create our own Stack class that does not allow the user to perform illegal operations.

    Implementing Stacks¶

    When implementing an abstract data type, you must first decide on the underlying representation. We will see three different implementations:

    1. An implementation using the LinkedList data structure we created in the last module (in the csc205 package).
    2. An implementation using arrays (in the jsjf package).
    3. An implementation using linked nodes (in the jsjf package).

    We will briefly discuss the three approaches above. See the example code and videos for each for a more detailed description.

    You may wonder why we bother implementing a stack (and later a queue) three different ways, or even at all since the Java API provides a stack for us. The reason we avoid the Java Stack is that it contains extraneous methods that allow our "stack" to be used like a Vector which is undesirable.

    Seeing different implementations has multiple benefits for you later on in your career:

    • You get the opportunity to work with code you did not write.
    • You learn more about how the data structures work so that you can make good choices about data structures.
    • You get to see different approaches to solving a problem.

    While you may never actually implement a stack again once you leave this class, the three benefits above will all make you a better programmer.

    The number of cells in the array is called its capacity. We canât change the capacity of an existing array. We should not have to worry too often about the stack reaching capacity, but when the top reaches the end of the array we will copy the array to an array of double the size.

    Analysis of Stack Methods¶

    All stack operations are O(1) for both linked and array based implementations. Since we know exactly where the top of the array is it is easy to find and modify. However, when using an array implementation, if the array is full it has to be resized which is an O(n) operation since we have to copy every element to the new array. Fortunately, resizing the array does not happen often - only after filling up the existing array. So while a worst case push operation could take linear (O(n)) time, that time averages out to O(1) time when spread out over all push operations (because if the size of the array is X it takes at least X push operations to fill the array).

    Queues¶

    A queue is a linear data structure whose elements are added in "first in, first out" (FIFO) order. We add elements to one end of the queue and remove them from the other. This has the effect of removing elements in the same order they are added to a queue.

    Queue Operations¶

    • enqueue(E element) adds an element to rear of the queue.
    • dequeue() removes & returns the top element from the queue.
    • first() returns the element at top of the queue.
    • size() returns the number of elements in the queue.
    • isEmpty() returns true if the queue is empty, false if it is not.

    Java API: Queues¶

    The Java API provides a Queue interface. The interface has two methods each for adding an element, removing an element, and returning the first element in the queue. The first set of enqueue/dequeue/first methods throws an exception if a problem occurs during the operation, whereas the second set of methods returns a value indicating an error (boolean for offer, null for poll/peek). This allows the programmer to decide how to handle exceptional cases.

    Queue Operation Throws Exception Returns Special Value
    enqueue add(element) offer(element)
    dequeue remove() poll()
    first element peek()

    Applications of Queues¶

    Queues have a wide variety of applications. Operating systems use queues extensively to manage access to a single resource, such as a disk, printer, or network. Any application that uses a buffer, such as watching an online video, uses a queue to store the data that has been downloaded before it is presented to the user. Queues can also be used to model real-world situations where there is a line, such as a call center or a ticket window.

    A Caesar cipher encodes a message by shifting each letter in a message by some constant k. For example, if k=5, then A becomes F, B becomes G, etc.

    Unfortunately, this is very easy to break (or fortunately if it is your enemies who are encoding messages this way).

    We can improve upon the Caesar cipher by changing the value of k depending on where the letter is in the message. We do this by using a repeating key: a series of integers that determine how much we shift a character.

    Given the repeating key {3 1 7 4 2}, we shift the first character by 3, the next by 1, the next by 7, and so on. When the key is exhausted, we start over at the beginning, so the 6th character would be shifted by 3, the 7th by 1, and so on. This is called the VigenÃ¨re cipher.

    Implementing Queues: Circular Array¶

    It may initially seem that we can use an array to implement a queue. However, since we add and remove from opposite ends, there is no way to avoid shifting the elements. If the front is at index 0, then we have to shift the elements to the left when we dequeue. conversely, if we set the rear to index 0, we have to shift all the elements to the right when we enqueue. Both of these are linear time operations - O(n) - which is inefficient.

    We can improve the performance by using a circular array. A circular array is an array that is treated as if it loops back around on itself. We consider index 0 to come after the last index. When we reach the last index, we use the modulus operator (%) to reset the index back to 0 like

    1
        rear = (rear + 1) % queue.count;

    Our CircularArrayQueue will have data members to keep track of the front and rear and the total size, and we resize the array when it is full.

    Analysis of Queue Methods¶

    All stack operations are O(1) for both linked & array based implementations, assuming our array implementation uses a circular array. Note that the same issues we had with resizing the array with stacks also occurs with our array queue implementation - a worst case push operation could take linear (O(n)) time though that time still averages out to O(1) time when spread out over all the enqueue operations.

    Queue Efficiency¶

    All operations in a Linked Queue are O(1). For an array implementation where we make index 0 the front of the queue, enqueue is O(1) but dequeue is O(n) since we have to shift the elements in the array. Circular arrays avoid the shift, so all operations for a circular array implementation are O(1). Therefore, like with a stack, all queue methods are O(1) given an ideal implementation (linked or circular queue).

    Linked versus Array-based Implementations¶

    Advantages to linked implementation:

    • Nodes only added as they are needed, so no wasted space in the data structure.
    • An efficient queue implementation is much simpler since the front/rear are not tied to a specific memory location.

    Advantages to array implementation:

    • No extra memory allocated for the node references.
    • Less wasteful if you know beforehand how big the stack or queue will need to be.
    • More efficient since there is no need to constantly allocate space for new elements and no garbage collection is required.
    • Arrays tend to be more optimizable by the compiler and JVM than linked lists.
